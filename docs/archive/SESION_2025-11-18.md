### 2. Error: URL Duplicada `/api/v1/api/v1/contracts/...`
**Archivo:** `frontend/src/components/contracts/MatchedCandidatesView.tsx:586`

**Causa:** Se agregaba `/api/v1/` cuando `api.post()` ya incluye baseURL

**Solución:** Removido prefijo duplicado
```typescript
// ANTES:
const response = await api.post(`/api/v1/contracts/landlord/contracts/${contractId}/approve_contract/`, {

// DESPUÉS:
const response = await api.post(`/contracts/landlord/contracts/${contractId}/approve_contract/`, {
```

---

### 3. Error: `TypeError: LandlordControlledContract() got unexpected keyword arguments: 'primary_party'`
**Archivo:** `contracts/landlord_contract_serializers.py:322`

**Causa:** El serializer intentaba establecer campo `primary_party` que no existe en el modelo `LandlordControlledContract`

**Modelo usa:** `landlord` (línea 67-72 de landlord_contract_models.py)

**Solución:** Cambiado el campo correcto
```python
# ANTES:
validated_data['primary_party'] = self.context['request'].user

# DESPUÉS:
validated_data['landlord'] = self.context['request'].user
```

## FLUJO CORRECTO DE CONTRATOS

### Etapas del Sistema
```
1. BORRADOR (LandlordControlledContract)
   └─ Estado: DRAFT
   └─ Ambas partes pueden revisar/modificar
   └─ NO está aprobado automáticamente

2. APROBACIÓN
   └─ Arrendador aprueba: landlord_approved = True
   └─ Arrendatario aprueba: tenant_approved = True
   └─ Ambos deben aprobar explícitamente

3. CONTRATO FINAL (Contract)
   └─ Solo después de aprobaciones
   └─ Inicia proceso biométrico
   └─ Modelo diferente para autenticación
```

### Modelos Duales
- **`LandlordControlledContract`**: Borradores y negociación
- **`Contract`**: Contrato final con biometría

**CRÍTICO:** Ambos registros deben existir, sincronizados
